Strings & slicing

Coming from Java's substring(begin, end), Python's s[start:stop:step] looks similar — but it's a tiny language of its own that also runs backwards, handles negative indices, and works on any sequence (lists, tuples, NumPy arrays). Learn it once here and you've learned indexing for the whole roadmap.

Two rulers, not one

The key mental model: a string isn't indexed by one set of numbers but two. Reading left-to-right, the first character is 0, the next 1, and so on. Reading right-to-left, the last character is -1, the one before it -2. Negative indices are just a shorthand for "count from the end" — s[-1] is always the last character, no matter how long the string is. That single trick removes a whole class of length - 1 arithmetic you'd write in Java.

The second idea is the half-open range. In s[start:stop] the start index is included but the stop index is excluded. It feels odd at first, but it's what makes s[:3] + s[3:] reconstruct the whole string with no overlap and no gap — the stop of one slice is the start of the next. Think of the indices as the gaps between characters, and a slice as "everything between gap start and gap stop."

The third number: step

s[start:stop:step] adds a stride. s[::2] takes every second character; s[::-1] — step of -1 — walks the string backwards, which is the famous one-liner for reversing a sequence. When you omit a bound, Python fills in a sensible default: for a forward step that's "the very start" and "the very end"; for a negative step the defaults flip. f-strings: formatting without the noise

The other half of "string fluency" is building strings from values. Forget "x = " + str(x) concatenation and forget String.format's positional %s placeholders — Python's f-string drops the expression right inside the literal, prefixed with f:

n = 3
f"{n} squared is {n*n}"  →  '3 squared is 9'
f"{n=}"  →  'n=3'  # the = is a debugging shortcut
f"{3.14159:.2f}"  →  '3.14'  # :.2f = 2 decimal places
f"{42:>6}"  →  '    42'  # >6 = right-align in 6 cols

Anything inside the {…} is evaluated as real Python — a variable, arithmetic, a method call. The optional : format spec controls decimals, padding, and alignment, which is exactly what you want when printing tidy tables of numbers later in the roadmap.

Why this matters beyond strings

Slicing isn't a string feature — it's a sequence feature. The identical [start:stop:step] grammar selects rows of a list, elements of a tuple, and (crucially) sub-arrays of a NumPy array in Phase 1. When you write X[:, 0] to grab the first column of a matrix, that's this same slice notation. Master it on a 7-character word now and the matrix version will feel obvious.

Takeaways: indices run 0,1,2… from the left and -1,-2… from the right. s[start:stop] is half-open — includes start, excludes stop — so adjacent slices tile perfectly. The third number is a step; s[::-1] reverses. Omitted bounds default to the ends. f-strings embed live expressions with f"{…}" and format with :spec.
